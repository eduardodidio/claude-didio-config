# Retrospective — F02

## What worked

- Wave dependency order (Wave 0 deps → Wave 1 parallel modules + diagrams → Wave 2 integration) prevented file conflicts and allowed T02/T03/T04 to run in parallel without stomping.
- `STATUS_STYLE` registry pattern is clean and extensible — adding a new status means editing one place.
- `forwardRef` + shadcn pattern for `Progress` meant the component slots naturally into any layout without prop-drilling a ref.
- `featureMap` (Map pre-computed in `useMemo`) is a better pattern than inline `features.find()` inside a render loop — eliminates O(n) lookups inside the JSX map.

## What to avoid

- **Aggregate/derived functions bypassing the canonical registry**: `getAggregateDot` was written with hardcoded Tailwind color strings (`'bg-red-500'`, `'bg-amber-500'`, `'bg-emerald-500'`) instead of delegating to `STATUS_STYLE`. When a module is introduced as a canonical map, all functions in that module must read from it. QA caught this and fixed it; but ideally the developer catches it during implementation.
- **Task test scenarios treated as optional**: T05 listed a "useMemo stability" test scenario that was not implemented. Task test scenario sections should be treated as a checklist, not illustrative examples.

## Patterns to repeat

- **Spy-based memo stability test**: using `vi.mock` to spy on the selector function and asserting call count is stable across re-renders is a clean, low-overhead way to validate `useMemo` correctness without React Profiler boilerplate.
- **`asChild` + Framer Motion `width` animation**: when Radix's `Indicator` uses `asChild`, the child owns its own layout; animating `width` instead of `transform` avoids conflicts with Radix's internal transform injection.
- **Pre-seed diagram directory in Wave 0 or T04**: having the diagram files scaffolded (even empty) in Wave 1 prevented Wave 2 tasks from needing to create them.

- `memory/agent-learnings/qa.md` — diff task scenario lists against test files; spy-based memo stability test pattern
